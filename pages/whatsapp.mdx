# WhatsApp

TextBubbles supports **WhatsApp** as a first-class send/receive channel alongside iMessage and SMS. WhatsApp uses the **same phone number** as your existing iMessage/SMS line — one paired WhatsApp account per number.

## Capabilities

| Feature | Outbound | Inbound |
|---|---|---|
| Plain text | ✅ | ✅ |
| Image (≤16 MB) | ✅ | ✅ |
| Video (≤64 MB) | ✅ | ✅ |
| Voice notes (PTT) via `attachments[].isAudioMessage: true` | ✅ | ✅ |
| Documents (≤100 MB) | ✅ | ✅ |
| Reactions | ✅ ([`POST /v1/messages/:id/reactions`](#reactions)) | ✅ via `message.reaction` webhook |
| Replies (quoted) via `replyTo` | ✅ | ✅ |
| Mentions in groups | ✅ | ✅ |
| Send to existing groups via `to: "...@g.us"` | ✅ | ✅ |
| Typing indicators | — | ✅ via `typing.indicator` webhook |
| Read receipts on outbound (delivered/read) | ✅ via `message.delivered` / `message.read` webhooks | n/a |
| iMessage-only fields (`effect`, `messageType: "carousel"`) | dropped | n/a |
| Group create / add / remove participants | not supported | not supported |

## Pairing a number

Pairing links your customer's WhatsApp account to a TextBubbles number. The phone number on the WhatsApp account **must match** the number on file — mismatched scans are rejected automatically.

### Via admin panel

1. Open **Instances** in the admin panel
2. Click **WhatsApp** on the row for the number you want to pair
3. Click **Enable WhatsApp** — a QR code appears
4. On the customer's phone: WhatsApp → Settings → Linked Devices → Link a Device → scan
5. The page polls automatically and flips to **Connected** once paired

### Via API

```bash
# 1. Create the session and start QR flow
curl -X POST -H "Authorization: Bearer $ADMIN_API_KEY" \
  https://api.textbubbles.com/v1/admin/instances/<instance_id>/whatsapp/enable

# 2. Poll for the QR string (raw — render with any QR library)
curl -H "Authorization: Bearer $ADMIN_API_KEY" \
  https://api.textbubbles.com/v1/admin/instances/<instance_id>/whatsapp/qr

# 3. Confirm the pair
curl -H "Authorization: Bearer $ADMIN_API_KEY" \
  https://api.textbubbles.com/v1/admin/instances/<instance_id>/whatsapp/status
```

### Pair-by-code (no QR)

If a QR display isn't available, request an 8-character code instead:

```bash
curl -X POST -H "Authorization: Bearer $ADMIN_API_KEY" -H "Content-Type: application/json" \
  -d '{"phoneNumber":"+15551234567"}' \
  https://api.textbubbles.com/v1/admin/instances/<instance_id>/whatsapp/pairing-code
```

The customer types the returned code into WhatsApp's "Link a Device" flow.

## Sending

See [Send Messages](/messages/send#whatsapp) for the full request shape. The `routing.preference` field decides which channel to try first; `to` is an E.164 phone number for individuals or a `...@g.us` JID for existing groups.

## Reactions

`POST /v1/messages/:id/reactions` works the same as for iMessage; the same `type` values apply on WhatsApp:

| `type` | WhatsApp emoji |
|---|---|
| `love` | ❤️ |
| `like` | 👍 |
| `dislike` | 👎 |
| `laugh` | 😂 |
| `emphasize` | ‼️ |
| `question` | ❓ |
| Any other string (e.g. `🔥`) | passed through verbatim |
| Negative form (e.g. `-love`) | removes the reaction |

```bash
curl -X POST -H "Authorization: Bearer $API_KEY" -H "Content-Type: application/json" \
  -d '{"type":"love"}' \
  https://api.textbubbles.com/v1/messages/msg_xxx/reactions
```

## Inbound

Inbound WhatsApp messages, reactions, typing indicators, and delivery/read updates flow through your registered webhook the same way iMessage events do. Each event includes `"channel": "whatsapp"` and a `customerId`.

Inbound media currently arrives base64-encoded inline in the webhook payload at `metadata.attachments[].dataBase64` for smaller files; very large media may arrive as a metadata-only entry. Treat the presence of `dataBase64` as the signal that the file is fully delivered.

## Limitations

- **One WhatsApp account per TextBubbles number.** Pairing uses the WhatsApp "Linked Devices" slot; revoking the link from the customer's phone disconnects the session and requires re-pairing.
- **Group operations.** Sending into existing groups (and replying / mentioning within them) is supported; creating groups or managing membership through the API is not.
- **iMessage-specific features** (`effect`, `messageType: "carousel"`) are silently dropped on a WhatsApp-only routing.

## Errors

| Error code | Meaning |
|---|---|
| `WHATSAPP_NOT_CONNECTED` | No paired WhatsApp session for this number. Pair via the admin panel. |
| `WHATSAPP_NUMBER_MISMATCH` | The scanned WhatsApp account's number doesn't match the registered TextBubbles number. |
| `WHATSAPP_RECIPIENT_NOT_REGISTERED` | The recipient phone number is not on WhatsApp. |
| `WHATSAPP_RATE_LIMITED` | Too many sends in a short window. Back off and retry. |
| `WHATSAPP_AUTH_REVOKED` | The session was revoked from the customer's phone. Re-pair to recover. |
| `WHATSAPP_UNSUPPORTED_FEATURE` | An iMessage-only field was sent to a WhatsApp-only routing. |
| `WHATSAPP_MEDIA_TOO_LARGE` | Attachment exceeds WhatsApp's size limits (16 MB image / 64 MB video / 100 MB document). |
| `WHATSAPP_REACTION_FAILED` | The reaction could not be delivered. |
